1 · Build pipeline

Multi-stage builds for Spring Boot

Separate compile, JRE assembly, and runtime concerns. The runtime stage should not carry Maven, Gradle, or a full JDK.

FROM eclipse-temurin:25-jdk AS build
WORKDIR /app
COPY . .
RUN ./mvnw package -DskipTests

FROM … AS jre-builder          # optional: jdeps + jlink
FROM gcr.io/distroless/base-debian13:nonroot
COPY --from=build /layers/… ./
ENTRYPOINT ["java", …, "JarLauncher"]
Name stages by purpose (build, jre-builder, runtime) — not by arbitrary numbers alone.
Layered JAR

Spring Boot layertools — cache-friendly runtime

Extract dependencies, loader, and application into separate directories. Docker layer cache reuses dependency layers when only app code changes.

FROM eclipse-temurin:25-jdk AS build
RUN java -Djarmode=layertools -jar target/*.jar extract --destination /layers

FROM … AS runtime
COPY --from=build /layers/dependencies/ ./
COPY --from=build /layers/spring-boot-loader/ ./
COPY --from=build /layers/snapshot-dependencies/ ./
COPY --from=build /layers/application/ ./
ENTRYPOINT ["java", "org.springframework.boot.loader.launch.JarLauncher"]

Requires Spring Boot 2.3+ layered JAR packaging in your build.

Cache mounts

Persist Maven / Gradle caches across builds

On ephemeral CI runners, mount a cache directory so dependency downloads are reused. Useful craft — measure on your pipeline if rebuild time matters.

# syntax=docker/dockerfile:1
RUN --mount=type=cache,sharing=locked,target=/root/.m2 \
    ./mvnw -B -q package -DskipTests

# Gradle:
RUN --mount=type=cache,target=/root/.gradle \
    ./gradlew build -x test
  • Enable BuildKit: export DOCKER_BUILDKIT=1
  • Pair with layered JAR so registry layers stay cache-friendly when only app code changes
Layer ordering

Put slow-changing files first

Anti-pattern

COPY . .
RUN ./mvnw package

Any source edit invalidates the package layer.

Better

COPY pom.xml mvnw .mvn/ ./
RUN ./mvnw dependency:go-offline
COPY src/ src/
RUN ./mvnw package -DskipTests

Dependencies cached until POM changes.

2 · Runtime image

Custom runtime with jlink

Use jdeps on your JAR to discover modules, then jlink to build a trimmed runtime — smaller than shipping a full vendor JRE.

FROM eclipse-temurin:25-jdk AS jre-builder
COPY --from=build /app/target/*.jar app.jar
RUN jdeps --print-module-deps app.jar > modules.txt
RUN jlink --add-modules $(cat modules.txt),java.logging,java.naming \
  --strip-debug --compress=2 --no-header-files --no-man-pages \
  --output /customjre

FROM gcr.io/distroless/base-debian13:nonroot
COPY --from=jre-builder /customjre /opt/java
ENV JAVA_HOME=/opt/java PATH="/opt/java/bin:$PATH"

Maintain a must-have module list for APIs jdeps misses (JNDI, logging, etc.).

Distroless runtime

Typical production tail

FROM gcr.io/distroless/base-debian13:nonroot@sha256:…
COPY --from=jre-builder /customjre /opt/java
COPY --from=build --chown=nonroot:nonroot /layers/dependencies/ ./
COPY --from=build --chown=nonroot:nonroot /layers/application/ ./
ENV JAVA_HOME=/opt/java PATH="/opt/java/bin:$PATH"
USER nonroot
ENTRYPOINT ["java", "-jar", "org.springframework.boot.loader.launch.JarLauncher"]

Distroless has no shell — use HEALTHCHECK on OS bases; rely on K8s probes on distroless.

3 · Cold start

Faster startup without native image

Spring Boot on the JVM can still improve time-to-readiness. Pick one primary lever and measure on your service.

AppCDS

Java 17+ shared class archive trained at build, loaded at runtime.

JEP 483 AOT cache

Java 24+ class-loading cache — needs custom jlink runtime.

Native image

Coldest start — longer builds and compatibility work.

AppCDS

Application Class Data Sharing

Train a .jsa archive during the build stage; ship it with the image. Works well with layered JAR layouts.

# Build stage — train archive
RUN cd /layers && java -XX:ArchiveClassesAtExit=/layers/app.jsa \
    -Dspring.context.exit=onRefresh \
    org.springframework.boot.loader.launch.JarLauncher || true

# Runtime stage
COPY --from=build /layers/app.jsa /app/app.jsa
ENTRYPOINT ["java", "-XX:SharedArchiveFile=/app/app.jsa", …, "JarLauncher"]

Available on Java 17+. Mutually exclusive with JEP 483 AOT cache in the same image — choose one path.

JEP 483

AOT class-loading cache (Java 24+)

Add an aot-trainer stage that runs the app once to produce app.aot, then load it at runtime.

FROM eclipse-temurin:25-jre AS aot-trainer
COPY --from=jre-builder /customjre /opt/java
COPY --from=build /layers/… ./
RUN java -XX:AOTCacheOutput=/app/app.aot … JarLauncher

# Runtime
COPY --from=aot-trainer /app/app.aot /app/app.aot
ENTRYPOINT ["java", "-XX:AOTCache=/app/app.aot", …, "JarLauncher"]

Best when tail latency (p95 startup) matters more than average — validate on your workload.

JVM in containers

Safe baseline flags

Use container-aware memory limits as hygiene — not a primary startup lever. Tune GC only after service-specific testing.

ENTRYPOINT ["java",
  "-XX:+UseContainerSupport",
  "-XX:MaxRAMPercentage=75",
  "-XX:+ExitOnOutOfMemoryError",
  "-Djava.io.tmpdir=/tmp",
  "org.springframework.boot.loader.launch.JarLauncher"]
  • Respect cgroup memory limits
  • Fail fast on OOM instead of limping
  • On distroless, enable GC logging — no shell to attach later
Native image

GraalVM native — when it fits

Native wins when

Cold start and footprint dominate — scale-to-zero, edge, burst traffic.

JVM wins when

Sustained throughput, fast iteration, reflection-heavy stacks.

FROM ghcr.io/graalvm/native-image-community:25 AS build
RUN ./mvnw -Pnative native:compile
FROM gcr.io/distroless/base-debian13:nonroot
COPY --from=build /app/target/nativeCompile/* /app/app
ENTRYPOINT ["/app/app"]

Build time is much longer — budget CI accordingly; test compatibility early.

Digest pins

Pin base images by digest

FROM eclipse-temurin:25-jdk@sha256:c2b7ea21… AS build
FROM gcr.io/distroless/base-debian13:nonroot@sha256:ab7554b6…
FROM debian:bookworm-slim@sha256:d5d3f9c2…

Rotate digests deliberately (Renovate, Dependabot, or internal catalog) — not floating :latest.

Anti-pattern

What not to ship

FROM ubuntu:latest
RUN apt-get update && apt-get install -y curl vim
COPY target/*.jar app.jar
USER root
ENTRYPOINT ["java", "-jar", "app.jar"]
  • Full OS + debug tools = large attack surface
  • Root + floating tag = supply chain and runtime risk
  • Prefer distroless non-root + digest pins + CI image scanning
Checklist

Monday-morning order

  1. Enforce security baseline — non-root, digest pins, SBOM
  2. Multi-stage + layered JAR; use cache mounts if CI rebuilds are slow
  3. Trim runtime with jlink; pick base image for your constraints
  4. Add one startup lever (AppCDS on 17+, or JEP 483 on 24+) only if you measure a win
  5. Consider native image only when cold start truly dominates

Why teams still skip it

What you actually get

Distroless + jlink, layered multi-stage, security controls as the default path — not optional extras.

The pain is delayed and diffuse. The effort looks optional — until security, cost, or cold start force the conversation.

Cost

Smaller image ≠ big registry bill

Sample Δ ≈ ~62 MB per full pull (130 → 68 MB). Storage is cheap; pulls × network path × services is where money shows up.

GB/month ≈ 0.06 × pulls × (1 − cache_hit)

Cost

When the Δ pays off

Saves more when

  • Many services share the fat-image pattern
  • High pull rate — deploys, CI, scale-out
  • NAT, cross-region, or internet egress
  • Cold runners / spot churn / new regions
  • Long tag retention without lifecycle cleanup

Saves less when

  • One service, rare deploys
  • Same-region free registry pulls
  • Warm node / layer caches
  • Tags already pruned aggressively

Order-of-magnitude only — plug in your pull count and $/GB. Security baseline and cold start still ride along on the same Dockerfile.

The hard part

Writing it once is easy. Keeping it true is not.

Hand-maintained craft

  • Copy the talk into one service
  • “We’ll harden in the next sprint”
  • Hero Dockerfile in a single repo

What actually happens

  • Next service ships Temurin + root again
  • Digests, SBOM, jlink never become habit
  • N services × silent drift

The checklist and decision map only stick if they become a default path you regenerate and gate in CI — not tribal knowledge.

Durable baseline

Strategy in git. Dockerfile as output. Drift fails CI.

1 · Strategy

.springdocker.toml — reviewable decisions (runtime, jlink, SBOM, pins…). The decision map as config.

2 · Artifact

Generated Dockerfile — multi-stage, layered JAR, jlink/distroless, hygiene. Owned, editable, in the PR.

3 · Gate

verify (+ config-drift) — merges fail when the Dockerfile stops matching the strategy.

That is what springdocker encodes from this talk — not a black-box image builder; a Dockerfile you still own.
