Methodology

What makes the numbers credible

  • One scenario per architectural choice (01–05 in the sample harness)
  • Multiple runs per variant — full profile for production decisions
  • Build time, image size, startup to readiness endpoint
  • p95 startup and regression checks in CI
springdocker benchmark generate --project-root .
springdocker benchmark run --profile full --runner-arg=--skip-native
springdocker benchmark analyze benchmarks/01-custom-jre-jlink/results/raw.csv
Caveats

Threats to validity

Host variance

Absolute ms change across laptops and CI runners.

Workload shape

Web API vs batch vs CPU-heavy changes the winner.

Competing goals

Cold start, throughput, and image size rarely align.

Use findings as decision guidance for your service — not universal laws.

JLink custom runtime

Decision 01

Use jlink + jdeps with curated must-have modules

jlink --add-modules java.base,java.logging,… \
  --strip-debug --compress=2 --output /customjre
  • Smaller runtime image vs vendor JRE on debian-slim or stock Temurin JRE image (~20–34% in our sample)
  • Merge must-have.txt modules jdeps misses (JNDI, logging, desktop APIs)
  • Skip only when compatibility or tooling blockers are proven on your app
use_jlink = true Performance · scenario 01

When ON — jre-builder stage + custom runtime

FROM eclipse-temurin:25-jdk AS jre-builder
RUN jdeps --print-module-deps app.jar > modules.txt
ARG MUSTHAVE_MODULES="java.desktop,java.logging,…"
RUN jlink --add-modules "$MODULES" … --output /jre/out
# runtime stage:
COPY --from=jre-builder /jre/out /opt/java
ENV JAVA_HOME=/opt/java

When OFF — vendor JRE copied in

FROM eclipse-temurin:25-jre AS vendor-jre
# runtime stage:
COPY --from=vendor-jre /opt/java/openjdk /opt/java
ENV JAVA_HOME=/opt/java
# distroless: gcr.io/distroless/java25-debian13 instead of base+jlink
Also: must_have_modules_file, jlink_baseline_modules → ARG MUSTHAVE_MODULES

Decision 03

Smallest image ≠ best production choice

distroless + jlink is the production-balanced profile default — strong size and startup in scenario 03 evidence. debian-slim still wins on build time when CI speed matters most.

Pick alpine

When every MB counts and musl compatibility is verified.

Pick distroless

When non-root hardening and smaller pulls matter more than build speed.

Decision 04

Choose by workload shape — benchmark both paths

Native wins when

Cold start and footprint dominate (scale-to-zero, edge, burst traffic).

JVM wins when

Sustained throughput, fast iteration, and reflection-heavy stacks matter.

springdocker generates a native scaffold Dockerfile; full native workflow is roadmap — not shipped yet.

recipe = "native-aot" Performance · scenario 04 · scaffold

When ON — GraalVM native compile

FROM ghcr.io/graalvm/native-image-community:25 AS build

FROM gcr.io/distroless/base-debian13:nonroot@sha256:…
COPY --from=build /app/target/nativeCompile/* /app/app
ENTRYPOINT ["/app/app"]

AppCDS class archive

VariantImageBuild avgStartup avgStartup p95
with-appcds167.89 MB614 ms1,262 ms1,350 ms

Java 17+ · works with layered JAR · mutually exclusive with JEP 483 AOT in the same image

# Train at build

# Load at runtime
java -XX:SharedArchiveFile=app.jsa -jar app.jar
Decision 05

Enable AppCDS when using layered JAR builds

  • Train archive in build stage; ship .jsa with the image
  • Lower startup latency without native-image complexity
  • Compare against end-to-end readiness — not isolated micro-benchmarks alone
enable_appcds = true Performance · scenario 05

When ON — train in build, load at runtime

RUN cd /layers && java -XX:ArchiveClassesAtExit=/layers/app.jsa \
    -Dspring.context.exit=onRefresh JarLauncher || true
COPY --from=build /layers/app.jsa /app/app.jsa
ENTRYPOINT ["java", "-XX:SharedArchiveFile=/app/app.jsa", …]

When OFF — no archive

# no ArchiveClassesAtExit RUN
# no COPY app.jsa
ENTRYPOINT ["java", …, "JarLauncher"]
Requires use_layered_jar = true · Java 17+ · mutually exclusive with JEP 483 AOT cache

Anti-pattern

What not to ship

FROM ubuntu:latest
RUN apt-get update && apt-get install -y curl vim
USER root
  • Huge attack surface · unnecessary packages · poor SBOM quality
  • Prefer distroless non-root + digest pins + verify gates in CI

What to do next

  2. Start from distroless + jlink (or run scenario 03 on your bases)
  3. Run one full-profile benchmark on a real service
  4. Adopt only changes with repeatable p95 or cost improvement
