# Changelog

Releases are generated by release-please and published from tagged builds.

**Versioning:** Dockly is a new product line. Shared release numbers start at **0.1.0** across the PyPI CLI, Maven plugin, and Gradle plugin (plugins use `0.1.0-SNAPSHOT` until the first Central publish). Entries below under `1.x` / `0.1.0` (springdocker era) are historical lineage only.

## Unreleased

### Changed

- **Aligned release versions:** CLI / Action docs / distribution examples use `0.1.0`; Maven and Gradle plugins use `0.1.0-SNAPSHOT` (release target `0.1.0`).
- **Rebrand to dockly (#3):** PyPI package / console script `dockly`, primary config `.dockly.toml`, env `DOCKLY_*`, GitHub Action, and Python import path `dockly`. Legacy `.springdocker.toml` and `SPRINGDOCKER_*` env vars are still accepted as a deprecation-window fallback.
- **Rebrand Java builder plugins (#4):** Maven artifact `dockly-maven-plugin` (goal prefix `dockly`), Gradle plugin id `io.github.mnafshin.dockly` (tasks `docklyGenerate` / `docklyVerify`). ADR 0010 POM/Gradle SSOT unchanged.
- **ProjectFacts model (#5):** structured detection with confidence + evidence (`dockly inspect` → `project_facts`); schema in `docs/project-facts.md`; fixtures for Spring and plain Java Maven/Gradle.
- **Strategy API (#6):** `ProjectFacts` + `Policy` → `StrategyPlan` with first-party `spring-boot-layered`, `spring-boot-jar`, and `plain-java` strategies; see `docs/strategies.md`.
- **Capability gating (#7):** documented matrix + generator/explain coverage for layered vs Spring jar vs plain Java; CLI `--use-layered-jar` / `--no-use-layered-jar`.
- **Plugin-seeded facts (#8):** implied `language=java` + build tool documented (`docs/plugin-facts.md`); detection fills the rest.
- **springdocker migration (#9):** `docs/migration-springdocker.md` shim policy; warn when loading `.springdocker.toml`.
- **Contributor strategy guide (#10):** `docs/contributing-strategies.md` + Go stub under `docs/examples/strategies/go/`.

### Migration (springdocker → dockly)

| Before | After |
|---|---|
| `pipx install springdocker` / CLI `springdocker` | `pipx install dockly` / CLI `dockly` |
| `.springdocker.toml` | `.dockly.toml` (legacy filename still loaded if primary missing) |
| `SPRINGDOCKER_*` | `DOCKLY_*` (legacy env names still honored) |
| `import springdocker` / `springdocker.*` entry points | `import dockly` / `dockly.*` (legacy entry-point groups still discovered) |
| Action input `springdocker-version` | `dockly-version` |
| `io.github.mnafshin:springdocker-maven-plugin` / `mvn springdocker:*` | `io.github.mnafshin:dockly-maven-plugin` / `mvn dockly:*` |
| Gradle id `io.github.mnafshin.springdocker` / `springdocker { }` / `springdockerGenerate` | `io.github.mnafshin.dockly` / `dockly { }` / `docklyGenerate` |
| `-Dspringdocker.*` Mojo properties | `-Ddockly.*` |

### Features

- `dockly setup` one-shot onboarding: detect project, write `.dockly.toml` (`production-balanced` by default), and generate `Dockerfile.generated` (`--interactive`, `--verify`, `--profile`, `--force` supported).
- `dockly setup --ci` / `--ci-only` writes `.github/workflows/dockerfile.yml` using the composite Action at `action/` (`mnafshin/dockly/action@v0` on the 0.x line).
- Pure-Java **Maven builder plugin** (`integrations/maven-plugin`) with POM SSOT: `generate`, `verify`, optional `export-config` (ADR 0010 / milestone Java builder).
- **Gradle twin** (`integrations/gradle-plugin`): `docklyGenerate` / `docklyVerify` with `build.gradle` SSOT.

### Documentation

- README / adopt / CLI quickstarts lead with `pipx install` + `dockly setup --ci`.
- GitHub Action usage documented in [`action/README.md`](action/README.md).
- Two-surface docs (Python CLI vs Java builder) in README, adopt, POSITIONING.
- Maven Central publishing runbook: `integrations/maven-plugin/PUBLISHING.md`.

### Notes

- An experimental Maven **CLI-wrapper** (subprocess to Python) was not merged; pure-Java plugin replaces that approach (#148).

## [1.2.0](https://github.com/mnafshin/dockly/compare/v1.1.1...v1.2.0) (2026-07-17)

### Features

- Java 17+ feature matrix (`java_features`) gates Dockerfile options by toolchain: floor is Java 17, JEP 483 AOT cache requires Java 24+, and unsupported combinations fail fast during configure / generate.
- Consumer smoke workflow against a pinned spring-petclinic clone (`scripts/consumer_smoke_petclinic.py` + CI job) exercises the full onboarding path on a real Spring Boot app.

### Changed

- Benchmark harness drops scenarios for multi-stage layout, BuildKit cache, and JVM container flags; remaining scenarios are renumbered **01–05** (`custom-jre-jlink`, `jep483-aot-cache`, `base-image-choice`, `native-benchmark`, `appcds`). CI regression baseline moves to `03-base-image-choice`.
- When `java_version` is omitted, dockly prefers the detected project Java and falls back to **17** (sample evidence project remains on Java 25).

### Documentation

- Docs restructured around production adoption (`docs/adopt.md`); positioning and presentation decks updated for the Java 17+ / sample-vs-CLI version story.
- Technical presentation assets added and refreshed for the reduced benchmark scenario set.

## [1.1.1](https://github.com/mnafshin/dockly/compare/v1.1.0...v1.1.1) (2026-07-12)

### Fixed

- Dockerfile generation detects which Gradle descriptor files exist and emits the correct `COPY` line for `.gradle` vs `.gradle.kts` projects (closes #130).
- After `dockly init`, the next-step hint now reads `next: dockly dockerfile generate` (closes #131).
- `SECURITY.md` supported versions updated for the 1.x line (closes #133).
- Generated Gradle Dockerfiles now select the boot JAR (excluding `*-plain.jar`) and copy it to a stable `build/libs/application.jar` path after `bootJar` (closes #135).
- Default digest pinning for generated Dockerfiles now includes `ubuntu:24.04` and `alpine:3.21` runtime stages (closes #132).
- `dockly verify` Trivy scans the Dockerfile path and its containing directory by default; use `--trivy-scan-project-root` to scan the full project root (closes #134).
- Recipe example Dockerfiles wired to `.dockly.toml` config (closes #128).

### Deprecated

- Benchmark legacy script delegation (`--use-legacy-scripts`, `DOCKLY_LEGACY_SCRIPTS`, and `[benchmark].legacy_scripts` in `.dockly.toml`) is deprecated and will be removed in **v2.0.0**. The internal benchmark runner is the default since 1.0.x; legacy mode now emits a warning when used (closes #137).

### CI

- The `supply-chain` job Trivy filesystem scan now **fails CI** on unfixed **CRITICAL** vulnerabilities (`exit-code: 1`). HIGH and below remain informational only in that job; use `dockly verify` locally or in your pipeline when you want HIGH+CRITICAL gates on Dockerfile build context (closes #139).

### Internal

- `configure_wizard.py` test coverage brought to 100% (closes #138).
- Mypy Phase 2 strict modules: `services.project_service`, `services.dockerfile_service`, `services.benchmark_service`, and `regression` (closes #136).

## [1.1.0](https://github.com/mnafshin/dockly/compare/v1.0.4...v1.1.0) (2026-06-20)

### Breaking changes

- Remove legacy Dockerfile generation delegation (`--use-legacy-scripts`, `wizard_args`, and `[dockerfile].legacy_scripts` on `dockerfile generate`). Use `dockly configure` for interactive setup and `dockly dockerfile generate` for CI. The sample `tools/dockerfile_wizard.py` script is retired (prints migration guidance).

### Features

- `dockly explain --config-aware` connects Dockerfile output to resolved `.dockly.toml` options, option sources (`default` / `project`), and drift detection vs `dockerfile generate`.
- `dockly verify --check-config-drift` validates config SSOT compliance (drift, embedded SBOM, non-root USER, configured JVM flags).

### Internal

- Ruff lint rules expanded to bugbear (`B`), pyupgrade (`UP`), and simplify (`SIM`) with baseline cleanup; gated in CI (closes #84).
- Mypy gradual strictness: Phase 1 module overrides, latent type fixes, typing roadmap (closes #90).
